Docker — por que contenedores
Un contenedor empaqueta el modelo, el codigo y todas sus dependencias en una unidad portatil. "Funciona en mi maquina" deja de ser un problema.
SIN DOCKER — EL PROBLEMA
El modelo usa Python 3.9 + scikit-learn 1.2. El servidor de produccion tiene Python 3.7 + scikit-learn 0.24. El modelo falla silenciosamente o da predicciones incorrectas.
CON DOCKER — LA SOLUCION
El contenedor lleva su propio Python 3.9 + scikit-learn 1.2. Corre identico en tu laptop, en staging y en produccion.

CI/CD — deploy automatico sin miedo
CI/CD (Continuous Integration / Continuous Deployment) automatiza el proceso: cada vez que el equipo sube codigo nuevo, los tests corren automaticamente y si pasan, el modelo se despliega solo.
1
Push a GitHub
El data scientist sube el nuevo modelo entrenado
2
Tests automaticos
Verifica que el modelo predice correctamente casos conocidos
3
Build del contenedor
Docker empaqueta el nuevo modelo
4
Deploy en cloud
Sin downtime: el trafico se migra gradualmente al nuevo modelo
✅ Beneficio clave: el equipo puede actualizar el modelo (reentrenar con datos nuevos) sin coordinacion manual. El modelo en produccion siempre es el mejor disponible.
Un modelo desplegado se degrada con el tiempo. El comportamiento de los clientes cambia, los datos de entrada cambian, el mundo cambia. Sin monitoring, nunca sabras cuando tu modelo deja de ser util.

Los 3 tipos de degradacion de un modelo
📉 DATA DRIFT
La distribucion de los datos de entrada cambia. El modelo fue entrenado con clientes de 2023, pero en 2024 llegan clientes con un perfil diferente (mas jovenes, mas digitales).
Ejemplo: el saldo promedio sube de S/.2,000 a S/.8,000 porque llegaron clientes premium. El modelo nunca vio ese rango en training.
▶ Detectar data drift
from scipy.stats import ks_2samp # Test Kolmogorov-Smirnov stat, p_value = ks_2samp( saldo_train, saldo_produccion ) if p_value < 0.05: alert("Data drift detectado en saldo")
🌿 CONCEPT DRIFT
La relacion entre features y target cambia. El patron de churn que el modelo aprendio ya no es valido porque el comportamiento de los clientes cambio.
Ejemplo: antes del COVID, baja actividad = churn inminente. Durante el COVID, baja actividad = normal. El modelo predice churn masivo erroneamente.
▶ Detectar concept drift
# Monitorear F1 semanal for week in recent_weeks: y_true = get_real_outcomes(week) y_pred = model.predict(get_features(week)) f1 = f1_score(y_true, y_pred) if f1 < threshold * 0.9: trigger_retrain()
⏱ MODEL DECAY
El modelo se vuelve obsoleto gradualmente aunque los datos no cambien bruscamente. Es el deterioro natural por el paso del tiempo y la acumulacion de cambios pequenos.
Regla practica: modelos de churn en retail: reentrenar cada 3-6 meses. Modelos de fraude: cada 2-4 semanas. Forecast de ventas: cada ciclo estacional.

Checklist de monitoring en produccion
METRICAS TECNICAS (IT/DS)
●Uptime de la API (>99.5%)
●Latencia p50/p95/p99 por endpoint
●Tasa de errores HTTP 5xx (<0.1%)
●Data drift en features criticas
●Distribucion de predicciones (si el 95% predice lo mismo, algo falla)
METRICAS DE NEGOCIO (MARKETING/PRODUCTO)
●Precision real del modelo: de los clientes marcados como churn, ¿cuantos realmente se fueron?
●Recall: de los que se fueron, ¿cuantos habiamos detectado?
●Revenue retenido por campanas activadas por el modelo
●Costo por falso positivo (cuanto gasta en retener a alguien que no iba a irse)
Como profesional de marketing digital, no necesitas saber programar para tomar decisiones correctas sobre modelos en produccion. Necesitas hacer las preguntas correctas, entender las metricas de negocio y saber cuando un modelo esta fallando aunque nadie te lo diga.
LO QUE DEBES PEDIR
Dashboard de metricas del modelo actualizado semanalmente. No basta con saber que "el modelo esta corriendo".
LO QUE DEBES EXIGIR
Un plan de reentrenamiento. Cada cuanto se actualiza el modelo con datos nuevos y quien es responsable.
LO QUE DEBES EVITAR
Confiar en el accuracy como unica metrica. Un modelo con 96% de accuracy puede tener Recall=0 en la clase que te importa.
🎯 CASO: MODELO DE CHURN EN PRODUCCION
Haz clic en cada pregunta para ver la respuesta y por que es importante hacerla:
❓ ¿Con que datos fue entrenado el modelo y hasta cuando llegan?
Critico para saber si el modelo conoce el comportamiento actual de tus clientes. Si fue entrenado con datos de hace 18 meses y el negocio cambio (nueva app, nuevo segmento, pandemia), el modelo puede estar prediciendo con un patron obsoleto. Exige saber la fecha de corte del training set.
❓ ¿Cuantos clientes marcamos como "churn probable" por semana?
Si la respuesta cambia drasticamente de una semana a otra sin razon de negocio, puede indicar data drift o un bug. El volumen de predicciones positivas debe ser estable o seguir la logica del negocio (mas en diciembre, menos en enero).
❓ De los clientes que el modelo marco como churn el mes pasado, ¿cuantos realmente se fueron?
Esto es la Precision real del modelo en produccion. Si el modelo marcaba 100 clientes/semana como churn y solo 20 se iban, la Precision es 20%. Cada intervencion de retencion con los otros 80 es costo innecesario (llamada, descuento). Si baja de un mes a otro, el modelo se esta degradando.
❓ ¿De los clientes que se fueron sin previo aviso, estaban en la lista del modelo?
Esto es el Recall. Si 50 clientes se fueron y el modelo solo habia detectado 20, el Recall es 40%. Hay 30 churns que el equipo no pudo retener porque el modelo no los vio. Esta metrica es la mas importante para el equipo de marketing: mide las oportunidades perdidas.
❓ ¿Con que threshold (umbral) esta operando el modelo?
El modelo devuelve una probabilidad (ej. 0.73 de churn). El threshold define a partir de que probabilidad se marca como "churn". Con threshold=0.5 el modelo es conservador; con 0.2 es mas agresivo (detecta mas pero con mas falsos positivos). Marketing debe decidir este umbral segun el costo de una llamada de retencion vs el costo de perder un cliente.
❓ ¿Hay un grupo de control sin intervencion para medir el impacto real del modelo?
Sin grupo de control, es imposible saber si el modelo esta generando valor o si los clientes se habrian quedado de todas formas. Un A/B test correcto: 80% de los clientes marcados como churn reciben la intervencion, 20% no. Comparas la tasa de churn entre ambos grupos. Esta es la unica forma de medir el ROI real del modelo.

TERMINOQUE SIGNIFICA EN SIMPLEPOR QUE TE IMPORTA
Endpoint / APILa "puerta" por donde tu sistema pide predicciones al modeloSi el endpoint cae, el modelo no funciona. Exige SLA de uptime.
LatenciaCuanto tarda el modelo en responder (en milisegundos)Una API con 3 segundos de latencia puede afectar la experiencia de usuario en tiempo real.
BatchProcesar muchos clientes a la vez en un horario fijoEl forecast semanal de ventas es batch. Los resultados llegan al dia siguiente, no en tiempo real.
Data driftLos datos de hoy son distintos a los del trainingSi hay drift, el modelo puede estar prediciendo mal sin que nadie lo sepa aun.
ReentrenamientoVolver a entrenar el modelo con datos mas recientesDebe tener un dueno, una frecuencia y un criterio de activacion.
Threshold / UmbralA partir de que probabilidad el modelo dice "si" o "no"Marketing decide este parametro segun el costo del error en cada direccion.
Shadow modeEl modelo nuevo corre en paralelo al viejo sin afectar produccionPermite comparar modelos sin riesgo antes de hacer el switch.
RollbackVolver al modelo anterior si el nuevo fallaExige que haya siempre un plan de rollback antes de cualquier deploy.
Los 3 casos — perspectiva de negocio y marketing
⚡ CHURN API — TIEMPO REAL
Que hace: cuando un cliente entra a la app, el sistema consulta la API y recibe la probabilidad de churn en <100ms.

Para marketing: trigger en tiempo real para mostrar oferta de retencion en el momento justo (banner, popup, notificacion push).

KPI clave: tasa de retencion de clientes marcados como churn que recibieron la intervencion vs grupo de control.
📦 FORECAST BATCH — SEMANAL
Que hace: cada domingo a las 2am, el modelo predice las ventas de las proximas 4 semanas para las 50 tiendas.

Para marketing: planificacion de campanas, presupuesto de pauta, stock de productos en promocion, asignacion de equipo.

KPI clave: MAPE real (error de prediccion vs ventas reales) por tienda y semana.
🔌 RECOMENDADOR — EMBED
Que hace: el modelo vive dentro de la app movil. Recomienda productos sin necesidad de internet, en <5ms.

Para marketing: personalizacion en zonas sin cobertura, catalogo offline, experiencia fluida sin dependencia de servidor.

KPI clave: CTR de recomendaciones, revenue atribuido al recomendador, tasa de conversion offline vs online.
