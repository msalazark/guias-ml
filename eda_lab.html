⚠️ Media vs Mediana — la prueba del skew:
Si media >> mediana → skew positivo (cola derecha). Si media << mediana → skew negativo. En este dataset: ticket (media 80.8 vs mediana 65.3) y saldo (4,068 vs 1,800) tienen skew positivo fuerte.

3. Outliers — deteccion con boxplot y regla IQR
🔍 Descubrir — ¿como detecto outliers?
Regla IQR:
Q1 = percentil 25 = S/.54.9
Q3 = percentil 75 = S/.83.7
IQR = Q3 - Q1 = S/.28.8
Limite superior = Q3 + 1.5×IQR = S/.127
Cualquier ticket > S/.127 es un outlier.

Resultado: 7 outliers encontrados con tickets entre S/.300 y S/.900.
TICKET PROMEDIO — 7 OUTLIERS
Q1=S/.54.9, Q3=S/.83.7, IQR=S/.28.8. Limite sup=S/.127. Los 7 outliers tienen tickets >300 — posibles clientes VIP o errores de registro.
⚠️ 3 preguntas antes de eliminar un outlier:
1. ¿Es un error de datos o un caso real?
2. ¿El outlier afecta mi metrica objetivo?
3. ¿Quiero que el modelo aprenda de ese caso extremo?
✅ Opciones: eliminar, capear (winsorize), transformar con log, o crear feature binaria "es_outlier".

Serie de ventas semanales — 52 semanas
🔍 ¿Que patrones busco en una serie de tiempo?
Los 4 componentes a identificar:

• Tendencia: ¿la serie sube, baja o es estable a largo plazo?
• Estacionalidad: ¿hay patrones que se repiten cada N periodos? (semanas, meses, anos)
• Ciclos: fluctuaciones de largo plazo no fijas (crisis, expansiones)
• Ruido: variaciones aleatorias que el modelo no puede predecir

En esta serie: tendencia creciente + estacionalidad semanal + pico en fin de ano.
🔍 ¿Que problemas pueden aparecer?
Problemas tipicos en series de tiempo:

• Gaps temporales: semanas sin datos (feriados, outages). ¿Hay que imputarlos?
• Cambios de regimen: la pandemia, un rebrand, un nuevo competidor → la serie cambia de comportamiento
• No-estacionariedad: media y varianza cambian en el tiempo → necesitas diferenciar
• Autocorrelacion: el valor de hoy depende del de ayer. ¿Cuantos lags son relevantes?
Descomposicion: tendencia + estacionalidad

La tendencia muestra el crecimiento sostenido de ~S/.200/semana. La componente estacional muestra el patron que se repite cada 12 semanas (trimestral). El residuo debe ser ruido blanco — si tiene patron, hay algo que el modelo no capturo.

El problema del desbalance — churn 4.3% vs no-churn 95.7%
🔍 ¿Por que es un problema?
El modelo "tonto" gana:

Si predices siempre "no churn", obtienes 95.7% de accuracy. Pero detectas 0 casos de churn reales. El Recall = 0, F1 = 0.

El accuracy es una metrica completamente inutil con clases desbalanceadas. Usa: Recall, Precision, F1, AUC-ROC, o F-beta score (con beta>1 si quieres priorizar Recall).
❌ Modelo naive (predice siempre 0):
Accuracy = 95.7% • Recall = 0% • F1 = 0
✅ Modelo util (detecta churns):
Accuracy = 82% • Recall = 74% • F1 = 0.68
🔍 ¿Como lo soluciono?
Estrategias para datos desbalanceados:

1. class_weight='balanced' en sklearn: penaliza mas los errores en la clase minoritaria
2. Oversampling (SMOTE): crea ejemplos sinteticos de la clase minoritaria
3. Undersampling: reduce la clase mayoritaria (pierde datos)
4. Ajustar el threshold: en vez de 0.5, usa 0.2 como punto de corte para churn
5. Cambiar la metrica: optimiza F1 o Recall, no accuracy
Distribucion de features por clase (churn vs no churn)
🔍 ¿Que busco aqui?
Si las distribuciones de churn y no-churn se solapan completamente para una feature, esa variable no discrimina entre clases. Si hay separacion visible, es una feature util. En nuestro caso: la actividad y antiguedad muestran diferencias entre churners y no-churners.

Reglas de interpretacion:

🔍 Correlacion espuria — ¿que es?
Correlacion espuria: dos variables correlacionan fuerte pero no hay causalidad real. Ejemplo clasico: ventas de helados correlaciona con ahogamientos (ambas suben en verano, pero una no causa la otra).

En datos de negocio: el numero de clicks correlaciona con revenue, pero puede ser porque ambas dependen del trafico total (variable oculta). Correlacion no implica causalidad.
